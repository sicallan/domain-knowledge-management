Why this exists

Your AI is only as good as the truth you can give it.

Indexing documents gives you plausible text. Enterprises — especially in regulated domains like Payments — need a model of what is actually true: the domain concepts, the rules, the invariants, and above all the decisions where regulation bites and business logic concentrates.

Not a document store

The document is the evidence; the inventory entry is the assertion; the graph connects assertions; views interpret; agents reason.

Decisions are the prize

The highest-value nodes. Make them visible, traceable and assessable — across every layer, end to end.

Every fact is evidenced

Each assertion links to its source provenance and carries bi-temporal versioning. Nothing is asserted without proof.

The mental model

Four layers — and everything maps to the truth layer.

L0

Strategic

Initiatives, value streams, stakeholder & value-impact maps — the why.

L1 — canonical

Pure Domain

DDD truth: domains, contexts, events, policies, invariants, decisions. Vendor-agnostic.

L2

Functional

Vendor products & project specs that claim to fulfil L1 (arrives Phase 3).

L3

Technical

Systems, services, runtime behaviour & operational evidence.

L1 is the spine. L2 and L3 map to it — which is exactly what makes cross-layer traceability possible (see Phase 2.5).

What we built

One pipeline. Extract once, load many.

The same real APIs that power the demo will power the product. Every boundary is a port/adapter — graph store, loaders, connectors, LLM gateway — so nothing is hard-wired.

sources→ connectors→ canonical docs→ extraction → JSONL→ GraphLoader→ graph→ Query Interface→ Views

Typed intermediate JSONL

Extraction emits schema-validated records; pluggable loaders fan them out to graph / vector / relational stores. Extraction never touches a store.

Two source formats, one registry

Markdown prose and a structured JSON export ingested through the same connector registry — adding a format was one line (OCP, demonstrated).

Adapter-agnostic graph

In-memory for dev/CI, Neo4j for integration — proven identical by port-parity contract tests before any DB lock-in.

Proof #1 · Phase 1

From prose to a navigable Domain Map — no manual modelling.

A real projection of the graph: subdomains → bounded contexts → counts and the cross-context relationships that emerge automatically. The gold hexagons are decisions.

Generated by the real View Projection Engine from payments-domain-map.json — the same DomainMapView the product UI will render.

Proof #2 · Phase 2

Runtime behaviour — with decisions fully traced.

The Card Authorisation Flow: ordered steps, the event that triggers it, the events & state-transitions it emits — and the decisions its steps invoke, each carrying the rules it evaluates, the data it consumes, the invariants that constrain it, and what it produces.

Exactly the evidence a compliance officer needs — generated from prose, not hand-drawn.

Proof #3 · Phase 2.5 — just shipped

One traversal: regulation → decision → realisation.

Cross-layer linking persists decision & regulatory edges with validated endpoints and makes them traversable both ways. Prove regulatory alignment and assess change blast-radius across L1 ↔ L3 in a single walk.

Guardrail (D-P2.5): at load time, edge endpoints are type-validated and cardinality-checked against one shared rule set. Wrong-typed, dangling or cardinality-breaching edges are quarantined for review and counted — never silently dropped, never committed as a dangling edge.
